# Spike: Native AOT vs SingleFile+R2R+Trim for WPF+WebView2 (.NET 8)

**Date:** 2026-05-12

## Goal
Pick the publishing strategy for Phase 6 packaging.

## Method
- Built a minimal WPF window hosting a WebView2 control that loads `about:blank` then NavigateToString.
- Attempted Native AOT publish.
- Attempted SingleFile + ReadyToRun + PublishTrimmed publish.
- For each: ran the produced exe, observed startup + WebView2 init, recorded size and time.
- Visual observation note: subagent shell cannot view the GUI. Methodology used was
  "soft pass" — launch the exe in background, wait ~10 s, confirm the process is
  still alive with no exception in captured stdout/stderr, then `Stop-Process` it.
  This validates that the runtime starts and the XAML graph (including the
  `WebView2` control's static init) loads without crashing; it does not prove
  the WebView2 surface visibly rendered "WebView2 OK". An interactive run by
  a human reviewer is needed to confirm the visual outcome.

## Native AOT result

- **Publish:** FAILED — error `NETSDK1168`:
  > 启用剪裁时，不支持或不推荐使用 WPF。请转到 https://aka.ms/dotnet-illink/wpf 以了解详细信息。

  Translation: "WPF is not supported or recommended when trimming is enabled."
  Native AOT on .NET 8 implicitly enables trimming (`PublishTrimmed=true`), and
  the .NET SDK rejects the WPF SDK + trimming combination at the
  `RuntimeIdentifierInference.targets` evaluation stage, before any IL processing
  starts. There is no documented workaround for WPF on .NET 8 — partial AOT for
  WPF is a future-roadmap item, not available in 8.0.420.
- **Exe size:** n/a
- **Cold start to WebView2 OK:** n/a — couldn't observe (no exe produced)
- **Runtime issues:** n/a — publish never ran

## SingleFile + R2R + Trim result

Two configurations were tried.

### Attempt A: `PublishSingleFile=true` + `PublishReadyToRun=true` + `PublishTrimmed=true`

- **Publish:** FAILED — same `NETSDK1168` (WPF rejects trimming, even partial).
  The error fires before MSBuild lets us specify `<TrimMode>partial</TrimMode>`,
  because the SDK validation gate predates trim-mode resolution. WPF + trim of
  any kind is not viable on .NET 8.

### Attempt B: `PublishSingleFile=true` + `PublishReadyToRun=true` + `PublishTrimmed=false`

This is the practical R2R fallback. Self-contained, no trim.

- **Publish:** SUCCESS — clean build, 0 warnings.
- **Exe size:** **171.1 MB** (`publish/r2r/MarkdownReader.exe`, single self-contained file
  with WPF + WebView2 + the .NET 8 runtime + native ARM/x64 dependencies extracted at
  first launch into a temp directory).
- **Cold start to WebView2 OK:** soft pass — exe launched, process alive for 10+ s,
  CPU ~0.5 s consumed (consistent with WebView2 EnsureCoreWebView2Async warmup),
  no exception in captured stdout. Visual render of "WebView2 OK" not confirmed by
  subagent; human spot-check still recommended for Task 6.x exit.
- **Runtime issues:** none observed.

## Decision

**Recommended for Phase 6:** **SingleFile + R2R (no trim)**

Reasoning: Native AOT is a non-starter for WPF on .NET 8 — the SDK refuses the
combination at the `NETSDK1168` validation gate, and there is no `TrimMode` knob
that lets us bypass it. SingleFile + ReadyToRun without trimming is the only
configuration that produced a runnable artifact. The 171 MB single-file size
is large but acceptable for an internal-distribution markdown reader; trimming
to reduce it is blocked at the SDK level until WPF gains trim support on a
later .NET. R2R precompilation still gives the cold-start win even without trim.

## Notes for Task 6.1

Embed in `MarkdownReader.csproj` (production publish profile or csproj
`PropertyGroup` guarded by `Condition="'$(Configuration)'=='Release'"`):

```xml
<PublishSingleFile>true</PublishSingleFile>
<PublishReadyToRun>true</PublishReadyToRun>
<PublishTrimmed>false</PublishTrimmed>
<IncludeNativeLibrariesForSelfExtract>true</IncludeNativeLibrariesForSelfExtract>
<SelfContained>true</SelfContained>
<RuntimeIdentifier>win-x64</RuntimeIdentifier>
```

Explicit notes:

- **Do not** set `PublishAot=true`. The WPF SDK rejects it with `NETSDK1168`.
- **Do not** set `PublishTrimmed=true` (same error). If a future .NET version
  lifts the WPF trim restriction, re-run this spike before flipping.
- `IncludeNativeLibrariesForSelfExtract=true` is required so the WebView2 native
  loader (`WebView2Loader.dll`) lives in the same extraction directory as the
  managed dlls at runtime. Without it, single-file extraction places native dlls
  in a sibling temp dir and `CoreWebView2Environment` can fail to locate them.
- `SelfContained=true` is required for a portable exe; the test machine doesn't
  need .NET 8 runtime installed. This is what drives the 171 MB size.
- Expected exe size: ~170 MB. If Task 6.1 sees materially different size, check
  the `RuntimeIdentifier` and `IncludeAllContentForSelfExtract` flags.
- Verify on a clean Windows machine without .NET 8 / Edge WebView2 Runtime to
  catch missing-dependency regressions before tagging Phase 6 done.
